# 首页恢复审计

## 结论

用户要恢复的是 iOS 文稿类 App 的完整首页结构：上半区提供品牌、用途说明和主要操作，下半区嵌入系统文件浏览器。当前仅显示系统文件浏览器、把新建入口压缩为顶部 `+` 的形态，不符合这次“改回去”的目标。

## 视觉证据

- 真机参考：`/Users/fze/Downloads/IMG_1834.PNG`
  - 上半区有大标题、简短说明、品牌视觉，以及“新建文稿”“扫描文稿”两个全宽按钮。
  - 下半区仍是系统文件浏览器，保留返回、网格/列表、更多、搜索和“最近项目 / 共享 / 浏览”。
- 仓库旧截图：`docs/screenshots/document-browser.jpg`
  - 已有“Margin + Create Document + 下方文件浏览器”的首页骨架。
  - 相比真机参考，它缺少用途说明、扫描入口，且主要操作仍为英文。
- 当前截图：`docs/localization-qa/current-document-browser.png`
  - 只剩下系统文件浏览器和顶部 `+`，上半区首页卡片完全消失，正是用户反馈的问题。

## 应恢复的关键结构

1. 顶部品牌区
   - 大标题使用 **Margin**，品牌名不翻译。
   - 可配 Margin 自己的 App 图标或品牌视觉；参考图中的“预览”和镜头图案属于 Apple Preview，不应复制。
   - 说明文案应为中文，例如：“选择一个文稿进行查看和编辑。”
2. 首页操作区
   - 主操作：“新建文稿”。
   - 次操作：“扫描文稿”。
   - 两项均为醒目的全宽按钮，并保留系统材质、圆角和动态字体适配。
3. 下方文件浏览区
   - 继续使用系统文档浏览器，显示本机、iCloud 与第三方文件提供商中的文稿。
   - 保留系统的返回、视图切换、更多、搜索，以及底部“最近项目 / 共享 / 浏览”导航。
   - 首页操作区和文件浏览区应同屏存在，不应让文件浏览器独占整个首屏。

## 中文与保留项

应汉化：

- `Create Document` → “新建文稿”
- `Scan Document` → “扫描文稿”
- `Choose a document to display and edit.` → “选择一个文稿进行查看和编辑。”
- 系统导航由简体中文本地化继续提供：“最近项目”“共享”“浏览”“搜索”等。

应保留：

- `Margin`：产品名。
- 用户文件名、文件夹名和第三方服务名：按原名显示，不强行翻译。
- `iPhone`、`iCloud` 等平台品牌词：沿用系统中文形式，如“我的 iPhone”“iCloud 云盘”。

不应照搬：

- 参考图中的“预览”、`PREVIEW 10x` 和镜头图案，它们是 Apple Preview 的品牌内容，不是 Margin 的通用系统文案。

## 与当前源码的对应关系

恢复前的 `Margin/App/DocumentBrowserViewController.swift` 曾设置 `allowsDocumentCreation = false`，并用顶部 `+` 自行创建文稿；这正是系统创建首页消失的原因。当前已删除这套 UIKit 入口并恢复 SwiftUI `DocumentGroup`。

“扫描文稿”是否能由目标系统 API 自动提供，需要在同一 iOS 版本真机上验证；如果系统 DocumentGroup 只恢复“新建文稿”，扫描入口需作为独立能力补齐，并确认扫描结果如何转换为本 App 支持的 Markdown/纯文本文稿，不能只做一个无结果按钮。

## 验收标准

- 冷启动时同时看到 Margin 品牌区、中文创建入口和下方文件浏览器。
- 不再出现英文 `Create Document` 或英文用途说明。
- 点击“新建文稿”能创建、打开、编辑并保存 Markdown 文稿。
- 若后续加入“扫描文稿”，必须提供真实可完成的扫描与导入流程，不能只有占位按钮。
- 现有文稿仍可从下方文件浏览器打开；iCloud、本机和第三方位置行为不倒退。
- 浅色/深色模式、真机/模拟器、动态字体下结构均可用。
